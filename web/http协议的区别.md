前端面试里一般重点比较：

```txt
HTTP/1.0
HTTP/1.1
HTTP/2
HTTP/3
```

核心演进方向是：

```txt
连接复用能力越来越强
传输效率越来越高
队头阻塞问题逐步减少
底层协议从 TCP 发展到 QUIC/UDP
```

------

# 1. HTTP/1.0

HTTP/1.0 比较早，特点是：

```txt
一次请求，一次 TCP 连接
请求完成后连接通常关闭
```

也就是说，浏览器请求 HTML、CSS、JS、图片时，可能要反复建立 TCP 连接。

问题是：

```txt
连接建立成本高
每次都要三次握手
性能较差
```

简单理解：

```txt
请求一个资源 → 建连接 → 请求 → 响应 → 断开
再请求一个资源 → 再建连接
```

------

# 2. HTTP/1.1

HTTP/1.1 是很重要的版本，主要改进是：

```txt
支持长连接
支持管道化
支持 Host 头
支持缓存控制
支持断点续传
```

最重要的是：**长连接**。

## 长连接

HTTP/1.1 默认开启：

```http
Connection: keep-alive
```

意思是：

```txt
一个 TCP 连接可以复用多次请求
```

这样就不用每次请求资源都重新建立 TCP 连接。

例如：

```txt
同一个连接：
请求 HTML
请求 CSS
请求 JS
请求图片
```

但是 HTTP/1.1 仍然有问题：**队头阻塞**。

------

## HTTP/1.1 的队头阻塞

在同一个 TCP 连接中，请求和响应基本还是按顺序处理。

如果前面的请求很慢：

```txt
请求 A 很慢
请求 B 已经准备好了
请求 C 也准备好了
```

B 和 C 也可能被 A 卡住。

所以浏览器通常会对同一个域名开多个 TCP 连接，比如 6 个左右，用来提高并发能力。

------

# 3. HTTP/2

HTTP/2 的核心改进是：**多路复用**。

## 多路复用

HTTP/2 可以在一个 TCP 连接里同时发送多个请求和响应。

```txt
一个 TCP 连接
同时传输 HTML、CSS、JS、图片等多个资源
```

它把数据拆成很多小帧：

```txt
请求 A 的帧
请求 B 的帧
请求 C 的帧
```

这些帧可以交错发送，最后再组装起来。

所以 HTTP/2 相比 HTTP/1.1：

```txt
减少 TCP 连接数量
提升并发能力
降低请求阻塞
```

HTTP/2 还有几个特点：

```txt
二进制传输
头部压缩
服务端推送
多路复用
```

------

## 但是 HTTP/2 仍然有 TCP 队头阻塞

注意一个面试细节：

```txt
HTTP/2 解决了 HTTP 层面的队头阻塞
但没有彻底解决 TCP 层面的队头阻塞
```

因为 HTTP/2 还是基于 TCP。

TCP 要保证数据按顺序到达，如果某个 TCP 包丢了，后面的包即使已经到了，也要等丢失的包重传。

所以 HTTP/2 在丢包场景下还是可能被 TCP 卡住。

------

# 4. HTTP/3

HTTP/3 最大变化是：

```txt
底层从 TCP 改成了 QUIC
QUIC 基于 UDP
```

也就是：

```txt
HTTP/1.1、HTTP/2：基于 TCP
HTTP/3：基于 QUIC，也就是基于 UDP
```

HTTP/3 的主要优势：

```txt
减少 TCP 队头阻塞
连接建立更快
弱网环境表现更好
支持连接迁移
```

------

## 连接迁移是什么意思？

比如你手机从 Wi-Fi 切换到 4G。

TCP 连接通常和：

```txt
IP + 端口
```

绑定。

如果网络切换，IP 变了，TCP 连接可能就断了。

QUIC 支持连接迁移，网络变化后仍然可以尽量保持连接不中断。

------

# 对比表

| 版本     | 底层协议 | 核心特点                         | 主要问题                   |
| -------- | -------- | -------------------------------- | -------------------------- |
| HTTP/1.0 | TCP      | 短连接                           | 每次请求都要建连接，性能差 |
| HTTP/1.1 | TCP      | 长连接、缓存、Host、断点续传     | HTTP 层队头阻塞            |
| HTTP/2   | TCP      | 多路复用、二进制帧、头部压缩     | 仍有 TCP 队头阻塞          |
| HTTP/3   | QUIC/UDP | 更快连接、减少队头阻塞、连接迁移 | 部署和兼容性要求更高       |

------

# 面试回答版本

可以这样说：

> HTTP/1.0 默认是短连接，每次请求资源都要重新建立 TCP 连接，性能较差。HTTP/1.1 默认支持长连接，可以复用 TCP 连接，并且增加了 Host 头、缓存控制、断点续传等能力，但它仍然存在队头阻塞问题。HTTP/2 引入了二进制分帧、多路复用和头部压缩，可以在一个 TCP 连接上并发传输多个请求，解决了 HTTP 层面的队头阻塞。不过 HTTP/2 仍然基于 TCP，所以在丢包时仍然可能出现 TCP 层面的队头阻塞。HTTP/3 则基于 QUIC，QUIC 底层使用 UDP，进一步减少队头阻塞，并且连接建立更快，弱网和网络切换场景下表现更好。

------

你重点记这条线：

```txt
HTTP/1.0：短连接
HTTP/1.1：长连接
HTTP/2：多路复用
HTTP/3：QUIC + UDP
```

小问题：**HTTP/2 已经有多路复用，为什么还需要 HTTP/3？**
